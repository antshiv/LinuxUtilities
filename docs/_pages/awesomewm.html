AwesomeWM Config

This AwesomeWM setup is the orchestration layer for LinuxUtilities. It binds keys, builds the status widgets, launches helper tools, and connects desktop events to the rest of the toolkit.

⌨
Current layout: rc.lua is the entry point and reusable logic now lives under linuxutils/*.lua. When you install the config, copy both the entrypoint and the module tree.

Installation

# Recommended: install the full user config tree from the repo
cd ~/Workspace/LinuxUtilities
make awesome-user-update

# Or update the system-wide config tree (uses sudo)
make awesome-update

# Reload AwesomeWM (no restart needed)
# Press Mod4+Ctrl+r  or:
awesome-client 'awesome.restart()'

The update targets install:

  • rc.lua as the entrypoint
  • linuxutils/common.lua for shared helpers
  • linuxutils/brightness.lua for brightness control

Config Tree

PathRole
rc.luaAwesomeWM entrypoint. Loads Awesome libraries, updates package.path, imports local modules, defines widgets, key bindings, rules, and signals.
linuxutils/common.luaShared helper functions used by the config: shell quoting, hostname lookup, command existence checks, compact label helpers, numeric clamping.
linuxutils/brightness.luaBrightness controller with backend fallbacks: brightnessctl, light, xbacklight, writable /sys/class/backlight/*, then xrandr.

System Stack

AwesomeWM is a user-space window manager. It does not talk directly to the Linux kernel to arrange windows. Instead, it receives events from the X server and calls out to user-space tools and daemons when it needs information or device changes.

What talks to what

  • Keyboard and device interrupts start in the kernel and input stack.
  • Xorg/X11 exposes those events to AwesomeWM.
  • awful.key callbacks in rc.lua turn those events into Lua actions.
  • Those actions may call helper commands like pactl, xrandr, nmcli, brightnessctl, or flameshot.

What this config adds

  • Media keys and brightness handling
  • Presenter controls for Gromit, spotlight, and dash drawing
  • Status widgets for audio, battery, network, Bluetooth, mail, and clock
  • Workflow launchers for the Linux control center and presentation tools

Example Event Flow

Brightness key example

  1. You press a laptop brightness key.
  2. X11 delivers an XF86MonBrightness* key event to AwesomeWM.
  3. rc.lua dispatches to linuxutils/brightness.lua.
  4. The brightness module tries hardware-friendly tools first, then falls back to xrandr software dimming if needed.
  5. A notification shows which backend actually handled the change.

How Module Loading Works

rc.lua first resolves the directory it was loaded from, then prepends that path to Lua's package.path. That matters because the same config must work both from the repository checkout and after installation into ~/.config/awesome or /etc/xdg/awesome.

local config_dir = resolve_config_dir()
package.path = table.concat({
    config_dir .. "?.lua",
    config_dir .. "?/init.lua",
    package.path,
}, ";")

local common = require("linuxutils.common")
local brightness = require("linuxutils.brightness")

This keeps the entrypoint small enough to reason about while still letting the config ship as a self-contained tree.

Writing AwesomeWM Lua

Common Awesome building blocks

  • awful.key for key bindings
  • awful.spawn and awful.spawn.easy_async_with_shell for external commands
  • naughty.notify for notifications
  • gears.timer for delayed refresh/retry behavior
  • wibox.widget for bar widgets and controls

Style of Lua used here

  • Keep shared logic in small modules that return tables.
  • Keep desktop actions as explicit local functions instead of anonymous shell blobs sprinkled everywhere.
  • Prefer one config entrypoint and small reusable modules over multiple competing rc.lua copies.
  • Validate with awesome -k before reloading.
local function volume_up()
    awful.spawn({ "pactl", "set-sink-volume", "@DEFAULT_SINK@", "+5%" }, false)
end

awful.key({}, "XF86AudioRaiseVolume", volume_up,
    { description = "increase volume", group = "media" })

Crash Recovery

If AwesomeWM crashes or fails to load the config:

  1. Press Ctrl+Alt+F2 to switch to a TTY
  2. Log in and check syntax: awesome -k -c ~/.config/awesome/rc.lua
  3. Restore the matching backup of rc.lua and, if needed, the matching linuxutils/ backup directory created by the Make targets.
  4. Press Ctrl+Alt+F7 to return to the graphical session
  5. Reload: awesome-client 'awesome.restart()'

Alternatively, the rescue-windows.sh script can recover floating windows if layouts break.
