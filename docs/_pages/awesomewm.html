AwesomeWM Config

This AwesomeWM setup is the orchestration layer for LinuxUtilities. It binds keys, builds the status widgets, launches helper tools, and connects desktop events to the rest of the toolkit.

⌨
Current layout: rc.lua is the entry point and reusable logic now lives under linuxutils/*.lua. When you install the config, copy both the entrypoint and the module tree.

Installation

# Recommended: install the full user config tree from the repo
cd ~/Workspace/LinuxUtilities
make awesome-user-update

# Or update the system-wide config tree (uses sudo)
make awesome-update

# Reload AwesomeWM (no restart needed)
# Press Mod4+Ctrl+r  or:
awesome-client 'awesome.restart()'

The update targets install:

  • rc.lua as the entrypoint
  • linuxutils/common.lua for shared helpers
  • linuxutils/brightness.lua for brightness control
  • linuxutils/icons.lua for palette-aware Bluetooth/network icon markup
  • linuxutils/audio.lua for volume/media control
  • linuxutils/presenter.lua for spotlight, Gromit, and presenter dash logic
  • linuxutils/launchers.lua for flameshot, control-center, and app launchers
  • linuxutils/calendar.lua for the large calendar/agenda popup and note preview
  • linuxutils/widgets.lua for wibar widget builders and status refresh
  • linuxutils/bindings.lua for root mouse bindings, global hotkeys, client keys, and client mouse bindings
  • icons/LinuxUtilitiesStatus/ for custom tray icon assets used by NetworkManager and Blueman

How The Modules Stitch Together

rc.lua
  -> resolve_config_dir()
  -> package.path += "<config dir>/?"
  -> require("linuxutils.common")
  -> require("linuxutils.brightness")
  -> require("linuxutils.audio")
  -> require("linuxutils.presenter")
  -> require("linuxutils.launchers")
  -> require("linuxutils.calendar")
  -> require("linuxutils.widgets")
     -> require("linuxutils.icons")
  -> require("linuxutils.bindings")
  -> construct controllers
  -> construct binding tables
  -> alias public actions for keybindings/mouse bindings
  -> build the wibar using widgets.lua
  -> install root/global/client bindings from bindings.lua

rc.lua now behaves like a composition root. The modules own the logic; the entrypoint owns wiring, layouts, bindings, and AwesomeWM lifecycle hooks.

Clock / calendar behavior

  • Left click the clock to toggle the large centered calendar popup.
  • Middle click opens an external calendar application.
  • Right click opens time/date preferences.
  • Mouse wheel on the clock browses previous/next months.
  • The popup includes font resize controls, a Close action, and a preview of today's markdown note.
  • Today's date is highlighted with a stronger accent so it stands out from the month grid.
  • Notes actions prefer Obsidian when installed; otherwise they fall back to opening the markdown note in vi inside a terminal.

System Stack

AwesomeWM is a user-space window manager. It does not talk directly to the Linux kernel to arrange windows. Instead, it receives events from the X server and calls out to user-space tools and daemons when it needs information or device changes.

What talks to what

  • Keyboard and device interrupts start in the kernel and input stack.
  • Xorg/X11 exposes those events to AwesomeWM.
  • awful.key callbacks in rc.lua turn those events into Lua actions.
  • Those actions may call helper commands like pactl, xrandr, nmcli, brightnessctl, or flameshot.

What this config adds

  • Media keys and brightness handling
  • Presenter controls for Gromit, spotlight, and dash drawing
  • Status widgets for audio, battery, network, Bluetooth, mail, and clock
  • Workflow launchers for the Linux control center and presentation tools

Example Event Flow

Brightness key example

  1. You press a laptop brightness key.
  2. X11 delivers an XF86MonBrightness* key event to AwesomeWM.
  3. rc.lua dispatches to linuxutils/brightness.lua.
  4. The brightness module tries hardware-friendly tools first, then falls back to xrandr software dimming if needed.
  5. A notification shows which backend actually handled the change.

How Module Loading Works

rc.lua first resolves the directory it was loaded from, then prepends that path to Lua's package.path. That matters because the same config must work both from the repository checkout and after installation into ~/.config/awesome or /etc/xdg/awesome.

local config_dir = resolve_config_dir()
package.path = table.concat({
    config_dir .. "?.lua",
    config_dir .. "?/init.lua",
    package.path,
}, ";")

local common = require("linuxutils.common")
local brightness = require("linuxutils.brightness")
local audio = require("linuxutils.audio")
local presenter = require("linuxutils.presenter")
local launchers = require("linuxutils.launchers")
local widgets = require("linuxutils.widgets")
local bindings = require("linuxutils.bindings")

After the local module requires, rc.lua constructs controllers and aliases their public methods into the names used later by keybindings and widget setup.

local audio_controller = audio.new({ ... })
local presenter_controller = presenter.new({ ... })
local launchers_controller = launchers.new({ ... })
local widgets_controller = widgets.new({ ... })
local binding_tables = bindings.build({ ... })

local volume_up = audio_controller.volume_up
local open_flameshot = launchers_controller.open_flameshot
local create_audio_widget = widgets_controller.create_audio_widget
root.buttons(binding_tables.root_buttons)

This keeps the entrypoint small enough to reason about while still letting the config ship as a self-contained tree.

Writing AwesomeWM Lua

Common Awesome building blocks

  • awful.key for key bindings
  • awful.spawn and awful.spawn.easy_async_with_shell for external commands
  • naughty.notify for notifications
  • gears.timer for delayed refresh/retry behavior
  • wibox.widget for bar widgets and controls

Style of Lua used here

  • Keep shared logic in small modules that return tables.
  • Keep desktop actions as explicit local functions instead of anonymous shell blobs sprinkled everywhere.
  • Prefer one config entrypoint and small reusable modules over multiple competing rc.lua copies.
  • Validate with awesome -k before reloading.
local function volume_up()
    awful.spawn({ "pactl", "set-sink-volume", "@DEFAULT_SINK@", "+5%" }, false)
end

awful.key({}, "XF86AudioRaiseVolume", volume_up,
    { description = "increase volume", group = "media" })

Widgets + Status Bar

The wibar is assembled in rc.lua and combines standard Awesome widgets with LinuxUtilities-specific status blocks. The right side hosts audio, battery, network, Bluetooth, keyboard layout, systray, host label, Thunderbird, clock, and layout indicators.

The network and Bluetooth widgets now render through linuxutils/icons.lua using light-colored Pango markup and are the primary status indicators by default. The tray applets are optional: if you set LINUXUTILS_AUTOSTART_STATUS_APPLETS=1 before starting AwesomeWM, nm-applet and blueman-applet are relaunched with GTK_ICON_THEME=LinuxUtilitiesStatus, Blueman is pushed to full-color mode by setting org.blueman.general symbolic-status-icons to false, and the bundled tray icon theme under icons/LinuxUtilitiesStatus/ overrides the icon names those applets request.

make awesome-test currently validates:

  • repo-copy syntax via awesome -k -c rc.lua
  • a runtime smoke wrapper via bash tests/awesomewm_runtime_smoke.sh for the controller and bindings smoke configs
  • a staged $XDG_CONFIG_HOME/awesome tree with the modular config layout
  • presence of critical bindings such as brightness, spotlight, gromit, and presenter dash
  • Makefile dry-run coverage for installing linuxutils/*.lua

The runtime smoke wrapper skips cleanly when a usable virtual X11 display is not available in the current shell.

Crash Recovery

If AwesomeWM crashes or fails to load the config:

  1. Press Ctrl+Alt+F2 to switch to a TTY
  2. Log in and check syntax: awesome -k -c ~/.config/awesome/rc.lua
  3. Restore the matching backup of rc.lua and, if needed, the matching linuxutils/ backup directory created by the Make targets.
  4. Press Ctrl+Alt+F7 to return to the graphical session
  5. Reload: awesome-client 'awesome.restart()'

Alternatively, the rescue-windows.sh script can recover floating windows if layouts break.
